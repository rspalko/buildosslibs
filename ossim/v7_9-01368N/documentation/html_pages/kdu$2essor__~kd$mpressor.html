kdu_stripe_decompressor::~kdu_stripe_decompressor

Java: Kdu_stripe_decompressor.Native_destroy

~kdu_stripe_decompressor( )

Java: public native Native_destroy( )

Go to class description.

Synopsis

Calls reset and finish do similar things, but finish does not clean up all physical memory. This destructor implicitly invokes the reset function to ensure that all memory has been deallocated.

The reset function (and hence this destructor) will work correctly if the object was used with a multi-threaded environment (i.e., non-NULL env argument was passed to start) and the processing was aborted, so long as you have been careful to either destroy the multi-threaded environment or invoke terminate or join on a non-NULL env_queue that was passed to start. The reset call (and hence this destructor) is also fine if finish has already been invoked since the last call to start.

If a call to start might not have been bracketed by a call to finish or reset already, you must be sure not to destroy the kdu_codestream object before this destructor is invoked, since the reset function that is implicitly called here attempts to close open tile interfaces that may still exist into the codestream.
